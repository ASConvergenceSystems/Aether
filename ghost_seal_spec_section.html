15. AETHER Ghost Seal Protocol (AGS)

15.1 Overview

The AETHER Ghost Seal Protocol (AGS) provides cryptographic authenticity, integrity, and confidentiality guarantees for the AETHER mesh. AGS is an open-standard ephemeral threshold signing protocol derived from the Ethereal Key Scheme (EKS) formalism defined in Aydelotte (2026).

Three properties are addressed:

PropertyMechanism
Beacon authenticityThreshold signature over canonical aether.json
Response confidentialityX25519 + XChaCha20-Poly1305 encryption
Registry integrityMerkle-sealed registry with signed root

AGS implementations MUST use only open-standard cryptographic primitives. No proprietary components are permitted in a conformant AGS implementation.

15.2 Definitions

TermDefinition
Ghost Key An assembled Ed25519 or ML-DSA-87 private key that exists only within a bounded ceremony window. Never persisted to storage.
Ceremony The atomic window during which shares are assembled, the Ghost Key is constructed, signing occurs, and the key is destroyed.
Share A fragment of the Ghost Key seed generated via Shamir (t,n) secret sharing, distributed across separate physical media.
Verification Key The long-lived public key corresponding to the Ghost Key. Embedded in aether.json. Never destroyed.
Seal The signature produced by the Ghost Key over the canonical beacon manifest. Embedded in aether.json as ghost_seal.signature.
Epoch A bounded ceremony time window. Recommended maximum: 300 seconds.
Canonical Form The normalized byte representation of aether.json used as input to the signing operation. Defined in §15.4.3.

15.3 Algorithm Suite

AGS defines two conformance tiers:

Operators SHOULD deploy AGS-PQ where liboqs is available. AGS-CLASSICAL provides no quantum forward secrecy and MUST NOT be used in threat environments with quantum-capable adversaries.

15.4 Key Ceremony

15.4.1 Setup (One-Time)

Performed once per beacon identity. Generates shares and the permanent verification key.

1. Generate random 32-byte seed S from a CSPRNG
2. Apply (t,n) Shamir secret sharing to S → shares [s₁ ... sₙ]
3. Derive Ghost Key:
       sk = HKDF-SHA3-512(
              ikm  = S,
              salt = "AETHER-GHOST-KEY-v1",
              info = beacon_id
            )
4. Generate keypair:  (sk, vk) = Ed25519.keygen(sk)
5. Distribute [s₁ ... sₙ] to n separate physical media
6. DESTROY S and sk from memory (secure zero, compiler-resistant)
7. Publish vk in aether.json as ghost_seal.verification_key

15.4.2 Signing Ceremony

Performed each time the beacon manifest is updated. Assembles the Ghost Key, signs, and destroys within a single atomic ceremony window. 

1. Load shares [sᵢ ... sₜ] from t distinct physical media
2. Reconstruct seed:  S = Shamir.reconstruct(shares)
3. Derive Ghost Key:  sk = HKDF-SHA3-512(S, salt, info=beacon_id)
4. Canonical manifest: m = canonical_serialize(aether.json)  [see §15.4.3]
5. Sign:              σ = Ed25519.sign(sk, m)
6. DESTROY S, sk from memory (secure zero, compiler-resistant)
7. Write σ to aether.json as ghost_seal.signature
8. Record ceremony timestamp as ghost_seal.signed_at

Ceremony window MUST close within T_ceremony (recommended: ≤ 300 seconds).
Shares MUST NOT remain assembled after step 6.

15.4.3 Canonical Serialization

The canonical form of aether.json is produced by serializing the following fields in this exact order. The ghost_seal.signature field is excluded from the input. Output MUST be UTF-8 encoded JSON with no insignificant whitespace. 

Field order:
  aether_version
  beacon_id
  node_url
  addressed_to
  human_visible
  machine_readable
  operator          (full object)
  confidentiality   (status field only)
  communication     (endpoint field only)
  mesh              (registry_url, specification_url)
  topics            (array, original order)
  status
  signal
  established

Implementations MUST produce identical bytes for identical input. Any deviation constitutes a non-conformant implementation.

15.5 Beacon Seal Schema

Add the following object to aether.json:

"ghost_seal": {
  "algorithm":        "Ed25519",
  "verification_key": "<hex-encoded 32-byte public key>",
  "signature":        "<hex-encoded 64-byte signature>",
  "signed_at":        "<ISO 8601 UTC timestamp>",
  "ceremony_epoch":   "<operator-assigned epoch identifier>",
  "share_threshold":  "<t-of-n, e.g. '3-of-5'>"
}

For AGS-PQ, algorithm is "ML-DSA-87" and key/signature lengths are per NIST FIPS 204.

15.6 Agent Verification

Agents SHOULD verify the Ghost Seal before acting on beacon content. Verification is REQUIRED before registering or relaying a beacon. 

1. Fetch aether.json from node_url
2. Extract:
     vk = ghost_seal.verification_key
     σ  = ghost_seal.signature
     alg = ghost_seal.algorithm
3. Remove ghost_seal.signature from manifest copy
4. Produce canonical serialization m per §15.4.3
5. Verify: Ed25519.verify(vk, m, σ)
6. If verification FAILS → beacon status = UNTRUSTED
   If verification PASSES → beacon status = VERIFIED

Unverified beacons MAY be read but responses SHOULD be withheld.

The /register endpoint SHOULD reject registration of beacons whose Ghost Seal fails verification. Registration of unsealed beacons (no ghost_seal field) is permitted but the node MUST be marked "seal_status": "UNSIGNED" in the registry.

15.7 Response Encryption

To protect agent responses from interception in transit, operators MAY publish an X25519 encryption public key. Agents SHOULD encrypt responses when this key is present.

Operator Setup

1. Generate X25519 keypair: (enc_sk, enc_pk)
2. Publish enc_pk in aether.json as encryption.public_key
3. Store enc_sk offline, separate from Ghost Key shares

aether.json addition

"encryption": {
  "algorithm":  "X25519+XChaCha20-Poly1305",
  "public_key": "<hex-encoded 32-byte X25519 public key>"
}

Agent Encryption Procedure

1. Generate ephemeral X25519 keypair: (e_sk, e_pk)
2. Compute shared secret: ss = X25519(e_sk, enc_pk)
3. Generate random 24-byte nonce
4. Derive symmetric key:
       k = HKDF-SHA3-512(
             ikm  = ss,
             salt = nonce,
             info = "AETHER-RESPONSE-v1"
           )
5. Encrypt: ct = XChaCha20-Poly1305.encrypt(k, nonce, plaintext)
6. Submit to response channel:
   {
     "encrypted": true,
     "e_pk":      "<hex>",
     "nonce":     "<hex>",
     "ct":        "<hex>"
   }

Operator Decryption

1. ss = X25519(enc_sk, e_pk)
2. k  = HKDF-SHA3-512(ss, nonce, "AETHER-RESPONSE-v1")
3. plaintext = XChaCha20-Poly1305.decrypt(k, nonce, ct)

15.8 Registry Seal

Registry operators SHOULD seal the registry with a Merkle root signature. This allows agents to verify no node entries have been added, removed, or modified since the registry was last sealed. 

Merkle leaf = SHA-256(beacon_id || node_url || registered)
Merkle tree = standard binary SHA-256 Merkle tree
              (duplicate last leaf for odd counts)
Merkle root = root of tree over all nodes in declaration order

aether-registry.json addition

"registry_seal": {
  "algorithm":        "Ed25519",
  "verification_key": "<hex>",
  "merkle_root":      "<SHA-256 Merkle root over node entries>",
  "signature":        "<hex signature over merkle_root bytes>",
  "sealed_at":        "<ISO 8601 UTC timestamp>",
  "node_count":       <integer>
}

Agent Verification

1. Fetch aether-registry.json
2. Recompute Merkle leaves and root from nodes array
3. Verify root == registry_seal.merkle_root
4. Verify Ed25519.verify(vk, merkle_root_bytes, σ)
5. If either check fails → registry = TAMPERED

15.9 Security Properties

PropertyAGS-CLASSICALAGS-PQ
Beacon forgery resistanceHard (DL assumption)Hard (MLWE assumption)
Ghost Key recoveryRequires t sharesRequires t shares
Quantum resistance (signing)NO — Shor breaks Ed25519YES — ML-DSA-87
Quantum resistance (encryption)NO — Shor breaks X25519YES — ML-KEM-1024
Forward secrecyYES — Ghost Key ephemeralYES
Response confidentialityHard (CDH assumption)Hard (MLWE assumption)
Registry tamper detectionYES — Merkle + signatureYES
Note on AGS-CLASSICAL and Quantum Adversaries

AGS-CLASSICAL provides no quantum forward secrecy on signing or encryption. A sufficiently capable quantum adversary running Shor's algorithm could recover the verification key's corresponding private key and forge seals retroactively. Operators in long-term deployments or high-threat environments MUST migrate to AGS-PQ. The AETHER mesh will formally deprecate AGS-CLASSICAL in v1.0.

15.10 Reference Implementation

A reference implementation of AGS-CLASSICAL is provided as aether_ghost_seal.py. It implements the full ceremony: share generation, key derivation, canonical serialization, signing, and secure destruction. 

# AGS-CLASSICAL dependencies
pip install cryptography    # Ed25519, X25519, HKDF, ChaCha20-Poly1305
# Shamir (t,n) is implemented natively — no additional dependency

# AGS-PQ additional dependency
pip install liboqs-python   # ML-DSA-87, ML-KEM-1024

The reference implementation is released under Apache 2.0 as part of the AETHER open standard. Operators are encouraged to audit and adapt it.
